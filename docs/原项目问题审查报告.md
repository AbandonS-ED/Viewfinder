# 原 iOS 项目 (NikonConnectIOS) 问题审查报告

> 审查对象：`D:\桌面\Nikon_connect\`（原 Swift 项目，70 个文件，~11000 行）
> 审查目的：**不是为了批判 iOS 代码质量，而是给 Viewfinder（Flutter 重写）找教训。**
>
> 每个问题附"行业级"写法 VS iOS 做法对比，和 Viewfinder 应采用的方案建议。

## 1. 协议层双重锁：actor + 手动 mutex

**严重度** 🔴
**来源**：`Services/PTPIPSession+Lifecycle.swift` 第 457-484 行、`Services/PTPIPSession.swift` 第 3 行

iOS 的 `PTPIPSession` 是 `actor`（Swift 内置序列化），但同时你自己写了一套 `commandChannelWaiters: [CheckedContinuation]` 配合 `isCommandChannelBusy` 做手动阻塞。

```swift
actor PTPIPSession {                  // ← 这是第一把锁
    var isCommandChannelBusy = false   // ← 第二把锁
    var commandChannelWaiters: [CheckedContinuation<Void, Never>] = []  // ← 等待队列

    func withExclusiveCommandChannelAccess<T>(_ op: ...) async throws -> T {
        await acquireCommandChannelAccess()  // ← 等待入队
        defer { releaseCommandChannelAccess() }  // ← 唤醒下一个
        return try await op()
    }
}
```

根因是 Swift actor 的隔离保证**只在同步入口点生效**——一旦方法在 `await commandConnection.receivePacket()` 处挂起，actor 锁就被释放，下一个进入 actor 的方法可以并发执行。`runEventMonitorLoop`（后台循环监听 event channel）也跟 `requestResponseOnly` / `requestDataIn` / `requestDataInToTemporaryFile`（三处都走同一个 mutex）共用同一 actor，所以必须手动加锁保证 command 通道串行化。

（早期版本曾推测 `eventMonitorTask` 可能读到 command 通道的响应包——这是错的，两条 socket 物理上是独立的，event channel 永远读不到 command 包。）

**建议**：
- Flutter 没有 actor —— 用 Future/Completer 更清楚
- 不要复刻"actor + 手动 mutex"的组合——拆成两个独立的小 service（command 串行化、event 独立循环），各自维护状态，不要共享 actor
- Phase 1 已经这样设计了（`_commandConnection` + `_eventConnection` + `_startEventMonitor` 分离）

## 2. AssetTraversal 685 行单文件堆叠 7 个状态

**严重度** 🟡
**来源**：`Services/PTPIPSession+AssetTraversal.swift`（685 行）

`PTPIPSession+AssetTraversal.swift` 是**全仓最厚的文件**。内含：

| 组件 | 行数 | 职责 |
|---|---|---|
| `loadAssetsPage` | ~93 | 分页核心，处理目录/文件/跳过三种分类 |
| `loadObjectHandles` | ~85 | 枚举 storage、尝试 Nikon 元数据、多策略 handle 获取 |
| `parseObjectInfo` | ~42 | 从 20+ 字段解析 PTP 数据集 |
| `photoKindFromObjectInfo` | ~36 | 根据扩展名 + objectFormat 判断类型 |
| `classifyObject` / `photoKind` | ~25 | 最终类型判定 |
| `objectHandleStrategies` | ~15 | 枚举策略生成 |
| `mergeDiscoveredHandles` / `mergeKindHint` | ~30 | 去重 |
| 状态结构体 | ~50 | `AssetTraversalState`(8字段) + `HandleDiscovery`(3) + `NikonObjectMetaData`(3) |

所有状态通过 `inout` + `try await` + `guard var traversal = assetTraversalState` 模式传递 —— 一个函数的副作用可能影响下一个函数，但参数是值传递的。

```swift
guard var traversal = assetTraversalState else { throw ... }
while pageAssets.count < pageSize, traversal.nextIndex < traversal.queue.count {
    let handle = traversal.queue[traversal.nextIndex]
    traversal.nextIndex += 1
    // 一堆操作可能修改 traversal
}
assetTraversalState = traversal  // ← 忘了这行？状态丢了
```

**建议**：
- Flutter 版应当用一个 `AssetTraversalService` class 封装状态，不要裸暴露 struct
- 拆出 `HandleDiscoveryStrategy`（策略模式）—— 按 storage / format 枚举实现不同策略
- `photoKindFromObjectInfo` 抽成纯函数（无状态依赖）

## 4. `UInt32(asset.remoteIdentifier)` 静默溢出

**严重度** 🟡
**来源**：`Services/ExperimentalNikonTransport.swift` 第 63 行、第 90 行

```swift
guard let handle = UInt32(asset.remoteIdentifier) else {
    throw CameraAppError.unsupportedOperation(...)
}
```

`remoteIdentifier` 是 `String`。`UInt32(...)` 在 Swift 里如果字符串表示的整数大于 `2^32 - 1` 返回 `nil`，**但如果字符串以 `-` 开头（负数）也会返回 `nil`**。实际上 remoteIdentifier 是 `String(info.handle)` 生成的（`PTPIPSession+AssetTraversal.swift:476`），handle 是 `UInt32`，所以一般不会出事。

但这是**隐式契约** —— 调用方必须知道 remoteIdentifier 是数字字符串才能正确 parse 回 handle。Phase 1 的 `_classifyObjectFormat` 也依赖这个契约。哪天任何地方改了 remoteIdentifier 的格式，parse 静默失败。

**建议**：
- `PhotoAsset.remoteIdentifier` 应当直接用 `int` 或 `UInt32` 而非 `String`
- 如果必须用 String（跨领域序列化），加类型包装 `AssetHandle` struct
- Phase 0 已经用 `String` 作为 `remoteIdentifier` —— 建议 Phase 2 换成 `int`

## 5. String 诊断日志是信息黑洞

**严重度** 🔴
**来源**：`Services/PTPIPSession+Lifecycle.swift` 第 496-499 行、`Services/PTPIPPrimitives.swift` 第 120-144 行

```swift
// PTPIPSession 内部
func recordDiagnostic(_ message: String) {
    diagnostics.append(message)
    AppLogger.transport.info("\(message, privacy: .public)")
}

// 外部调用方
func consumeDiagnostics() async -> [String] {
    let current = diagnostics
    diagnostics.removeAll()
    return current
}
```

所有异常信息在 `PTPIPError.errorDescription` 阶段就压成中文长字符串（例：`"相机返回了错误响应：0x2005。"`），然后以 String 形式传到日志和诊断。调用方无法程序化区分：
- 是 timeout（重试可恢复）？
- 还是 `operationNotSupported`（不可恢复）？
- 还是 `deviceBusy`（等等再试）？

`ExperimentalNikonTransport.mapError` 又把所有 PTPIPError 压成 `CameraAppError.networkProbeFailed`（第 165-173 行）—— 信息完全丢了。

**建议**：
- `CameraAppError` 已经用 sealed class 解决 —— 错误类型可 pattern match
- Phase 2 应当把诊断日志改成结构化 `DiagnosticEntry { timestamp; level; code; message; }`
- 给 `PTPIPError` 加 `responseCode` getter，让外部代码不用 pattern match 也能按 code 区分（Viewfinder Phase 1 已加）

## 6. `CameraTransport` 协议扩展和默认实现混用

**严重度** 🟡
**来源**：`Services/CameraTransport.swift` 第 17-68 行

```swift
protocol CameraTransport: Sendable {
    func downloadThumbnail(...) async throws -> Data?   // ← 抽象方法
    func downloadAssetToTemporaryFile(...) async throws -> URL  // ← 抽象方法
}

extension CameraTransport {
    // 默认实现：返回 nil（"不支持"）
    func downloadThumbnail(...) async throws -> Data? { nil }

    // 默认实现：调用 downloadAsset + 写文件
    func downloadAssetToTemporaryFile(...) async throws -> URL {
        let data = try await downloadAsset(...)
        // ... 写文件
    }
}
```

两个问题：
1. `downloadThumbnail` 的默认实现返回 `nil`，但调用方（`AssetThumbnailService`）把它当"没缩略图"而不是"该操作不支持"——语义模糊
2. `downloadAssetToTemporaryFile` 的默认实现已经包含完整逻辑，但 `ExperimentalNikonTransport` 仍然覆写了它（第 77-120 行），意味着协议默认实现从未被测试过

**建议**：
- 协议里要么只声明不实现，要么提供确定的默认实现。不要 `nil` 表示"空值"和"不支持"
- Flutter 版更清晰：`FutureOr<Uint8List?>` 只在真正可选时返回 null；不支持直接抛异常

## 7. `DownloadStore` 全量序列化每次写

**严重度** 🟡
**来源**：`Services/DownloadStore.swift` 第 55-120 行

`upsertDownloadJob` / `removeDownloadJobs` / `markInterruptedRunningJobs` 每个方法的 return 都是整个 `DownloadQueueState`：

```swift
func upsertDownloadJob(...) throws -> DownloadQueueState {
    var state = try loadQueueState()      // ← 读文件
    state.jobs[...] = job
    try saveQueueState(state)             // ← 写文件（全量序列化）
    return state                          // ← 返回全量
}
```

200 条下载记录时，每个小操作（更新一条进度）都序列化并写入整个 state（~100KB+）。虽然 iOS 是写 UserDefaults，但也有 IO 开销。

**建议**：
- 小变更只写变更的数据，读时重建全量
- 或引入 `@Published` 模式在内存中维护 state，写文件异步 buffer
- Phase 3（下载）实现时注意 —— 频发进度更新不要每次都全量 json

## 8. `AssetThumbnailService` 6 个字典冗余

**严重度** 🟡
**来源**：`Services/AssetThumbnailService.swift` 第 57-63 行、第 160-221 行

```swift
private var cachedThumbnailData: [String: Data] = []
private var inFlightTasks: [String: Task<Data?, Never>] = []
private var unavailableIdentifiers = Set<String>()
private var cachedPreviewData: [String: Data] = []
private var previewTasks: [String: Task<Data?, Never>] = []
private var unavailablePreviewIdentifiers = Set<String>()
```

thumbnail 和 preview 完全对称的逻辑，但用两套字段存储。所有存取方法都是 6 个 `switch kind`。

```swift
private func storeDataInMemory(_ data: Data, for cacheKey: String, kind: DerivativeKind) {
    switch kind {
    case .thumbnail: cachedThumbnailData[cacheKey] = data
    case .preview:   cachedPreviewData[cacheKey] = data
    }
}
```

这种代码在编译器层面是安全的（switch 穷举所有 case），但在添加第三种 derivative（比如 `posterFrame`）时代码膨胀为 9 个 switch。

**建议**：
- 用 `EnumMap<DerivativeKind, CacheStore>` 而不是 6 个独立字段
- Flutter 可以 `class CacheStore { Map<String, Data> cache; Map<String, Task> tasks; Set<String> unavailable; }`
- 然后在 `AssetThumbnailService` 里 `final stores = {for (var k in DerivativeKind.values) k: CacheStore()};`

## 11. `disconnect()` 同步清 coordinator + 异步关 transport 导致下载误判中断

**严重度** 🔴
**来源**：`Features/ConnectionSetup/ConnectionViewModel.swift` 第 112-128 行

`ConnectionViewModel.disconnect()` 的实现是：

```swift
func disconnect() {
    sessionCoordinator.clearSession()   // ← 同步清：activeTransport = nil
    Task { try? await transport.disconnect() }  // ← 异步关 socket
}
```

`sessionCoordinator.clearSession()` 同步执行后，`activeTransport` 立刻变 nil。然后 `Task { }` 才在后台慢慢关 socket。

中间这段窗口（几十毫秒到上百毫秒），`DownloadManagerViewModel.runQueue`（第 371-372 行）恰好在做以下检查：

```swift
guard let transport = sessionCoordinator.activeTransport else {
    markInterrupted(reason: "请重新连接 Nikon 相机后继续下载")
    return
}
```

它看到 `activeTransport == nil`，就把当前下载任务**标记为中断**——而实际上 transport 这一瞬间还没断，用户没出错，连接还活着。这就是用户反映的"我刚点断开，正在下载的任务就被错误地标成中断"的根因。

**为什么 `setActiveSession` 不是更严重的那个**：审计早期版本把 `setActiveSession` 当主风险，但实际 `setActiveSession` 当前**只在 `ConnectionViewModel.connect()` 调用一次**，UI 层按钮也互斥，场景上是理论风险。

**建议**：
- `disconnect()` 必须**先停下载队列 → 关 socket → 才清 coordinator**，顺序反过来
- 或者改 coordinator 状态为 `disconnecting`，跑队列遇到这个状态时**挂起等待**而不是直接标中断
- Viewfinder 的 Riverpod 写法：把 coordinator 拆成 `cameraSessionProvider`（StateNotifier），UI 监听其状态，下载 runner 用 `select` 等待状态恢复

## 13. PTPIPPrimitives 用 `enum` 做命名空间

**严重度** 🟢
**来源**：`Services/PTPIPPrimitives.swift` 第 147-150 行

```swift
enum PTPIPBinary {
    static let protocolVersion: UInt32 = 0x0001_0000
    static let defaultFriendlyName = "NikonConnectIOS"
}
```

Swift 里 `enum` 不能实例化、只有静态成员，常用于做命名空间（替代 `class`/`struct`）。但所有 `encode*` / `decode*` 静态方法也挂在这个 enum 下。这在语义上是 **"命名空间 = class"** 的 hack。

```swift
extension PTPIPBinary {
    static func encodePacket(...) -> Data { ... }
    static func decodePacketType(_ rawValue: UInt32) throws -> PTPIPPacketType { ... }
}
```

Dart 不允许 enum 挂静态方法 —— 所以 Phase 1 计划正确地将它改为 `class PTPIPBinary { PTPIPBinary._(); }`。这是一个**误用了语言特性**的例子，Flutter 版不用学。

## 15. `group.next()!` 强解包 TaskGroup

**严重度** 🔴
**来源**：`Services/PTPIPTCPConnection.swift` 第 163 行

```swift
let result = try await group.next()!
```

`ThrowingTaskGroup.next()` 返回 `ChildTaskResult?`，这里用了 `!` 强解包。在当前代码下，子任务一定会产出值，所以**实践上不会 crash**。但：

- 如果任何一个子任务在产出值之前抛异常，`group.next()` 的行为依赖 Swift runtime 实现（按文档说明会重新抛出，但跟 `!` 配合时语义模糊）
- 一旦未来重构添加了第三个分支或某个分支改成可选产出，这里直接 crash
- 类似 `force-unwrap` 在协议层是最危险的——直接发生在 TCP IO 路径

**建议**：
- 改 `guard let result = try await group.next() else { throw PTPIPError.sessionUnavailable }()`
- Viewfinder Phase 1 完全无此模式——所有 nullable 显式处理

## 16. 事件监控循环非超时异常退出后 Task 引用不清

**严重度** 🟡
**来源**：`Services/PTPIPSession+Lifecycle.swift` 第 411-426 行

`runEventMonitorLoop` 捕获到非 timeout 的 `PTPIPError`（例如 `connectionClosed`）后：

`eventMonitorTask` 这个 Task 引用**没有清空**。结果：

## 17. `consumeDiagnostics()` 是 on-demand 调用，长下载期间错误日志静默丢失

**严重度** 🟡
**来源**：`Services/ExperimentalNikonTransport.swift` 第 154-163 行 + 调用方 `ConnectionViewModel.swift` / `GalleryViewModel.swift`

```swift
func consumeDiagnostics() async -> [String] {
    let current = pendingDiagnostics + (await session?.consumeDiagnostics() ?? [])
    pendingDiagnostics.removeAll()
    return current
}
```

`consumeDiagnostics()` 是被动 drain —— 调用方 `appendTransportDiagnostics()` 只在 UI 状态变化时偶尔拉一次（`ConnectionViewModel.swift:202`、`GalleryViewModel.swift:140`）。

**后果**：

- 长下载（10+ 分钟）过程中如果中间出错，diagnostic 累计在 `pendingDiagnostics`
- 如果用户停在某个页面不切换，diagnostic 永远不显示
- `recordDiagnostic` 只追加到内存 + 打日志，UI 不知道有错

**建议**：
- 改 push 模型：`recordDiagnostic` 直接 emit 到 Stream / Notifier，UI 用 `StreamBuilder` / `ref.listen` 订阅
- 或者后台定时 drain（每 5 秒调一次 `consumeDiagnostics`）
- Viewfinder Phase 1 的 `_pendingDiagnostics` 是同款设计——Phase 2 改 push 模型

## 18. DownloadStore 进度更新频率×全量序列化 = 磁盘 IO 风暴

**严重度** 🟡

func persistProgressIfNeeded() {
    guard Date().timeIntervalSince(lastPersistAt) > 0.75 else { return }

## 值得 iOS 项目骄傲的地方（应该保留的）

不是全部都不好，以下 iOS 设计是 Flutter 版应该学习的：

| 好设计 | 保留理由 |
|---|---|
| **PTPIPPrimitives 的编解码封装** | 数据读取器 + 编码器分开，小端序统一，经测试验证 |
| **actor 隔离 PTPIPSession 状态** | session 级状态（transactionID / deviceInfo / isOpen）不会泄露脏数据 |
| **AdaptiveChunkController 分块自适应** | bandwidth-aware 的 chunk 调节（连续成功则加倍、失败则减半）值得翻译到 Flutter |
| **DownloadRecord + DownloadQueueState + @Published** | state 的三层隔离（record / queue / UI）可在 Riverpod 中复用 |
| **handle dedup by `seenHandles: Set<UInt32>`** | 多策略 handle discovery 的去重逻辑是 Nikon 协议坑 —— Flutter 要保留 |
| **Dual connection 架构** | 虽然实现有瑕疵，但 command/event 拆分是对的 —— Phase 1 已经保留 |
| **PTPDataType Parser** | `PTPDataReader.readUInt16()` / `readPTPString()` / `readUTF16NullTerminatedString()` 等编解码原语质量好，Flutter 直接翻译 |
| **缩略图生成降级链** | `cameraThumbnail → fileBacked → downloadAsset → fallback thumbnail` 的递归策略是合理的，Flutter 版可以复用 |
| **中间状态视图模型（isShowingGlobalActivity）** | UI 层不做逻辑、只做映射 —— 保持了视图的干净 |
| **scenePhase 感知** | `onChange(of: scenePhase)` → `handleScenePhaseChange` 让 VM 感知前台/后台切换，Flutter 版需要 `WidgetsBindingObserver` |
